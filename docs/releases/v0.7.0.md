# v0.7.0 release notes

Release status: release candidate for `v0.7.0`

## status summary

| Area | Status | Notes |
| --- | --- | --- |
| Async cancellation lifecycle evidence | released | Deterministic fixtures and tests cover the supported cancellation and late-result-drop cases. |
| ROS-level pre-emption fixture | released | Thin fixture that reuses canonical `host_action_invalid` and `fallback_used` events. |
| Lisp DSL round-trip and hash evidence | experimental | Evidence baseline for the Lisp-as-DSL claim. |
| Generated-fragment rejection fixtures | experimental | Deterministic validation fixtures only; generated fragments are not executable by default. |
| Host capability bundle contracts | contract-only | `cap.motion.v1` and `cap.perception.scene.v1` remain host-owned contracts. |
| `cap.echo.v1` registry smoke path | released | Deterministic registry/API fixture only. |
| `cap_call_start` / `cap_call_end` | planned | Chosen future event shape for real runtime-affecting capability adapters. |
| Production VLA providers | planned | Lifecycle infrastructure exists, but real providers are not released here. |
| Nav2/MoveIt adapters | planned | Roadmap work unless a later release note says otherwise. |

## what is stable in v0.7.0

- runtime contract and canonical event schema remain on the same external versions:

    - runtime contract: `1.0.0`
    - event schema: `mbt.evt.v1`

- async cancellation and timeout evidence is now covered by deterministic tests and fixtures for:

    - cancellation before start
    - cancellation while running
    - cancellation after timeout
    - repeated cancellation
    - late completion after cancellation

- canonical logs cover the required lifecycle outcomes for the supported async cases:

    - cancellation request and acknowledgement
    - timeout
    - fallback
    - late-result drop

- a thin ROS-level pre-emption fixture reuses the same canonical runtime events instead of introducing a ROS-specific cancellation model:

    - `host_action_invalid`
    - `fallback_used`

- the curated benchmark bundle path is documented in [runtime performance](../internals/runtime-performance.md), including interpretation limits and the need to verify the full `B8` group artefacts.

- release artefacts publish CMake package version `0.7.0`

## Lisp DSL defensibility baseline

`v0.7.0` makes the Lisp choice explicit as a technical representation choice rather than a style preference.

The documentation now states that Lisp is used as compact structured BT data for:

- authored BT source
- generated BT fragments
- validation input
- canonical serialisation
- replay artefacts
- future safe tick-boundary installation

The main entry point is [why Lisp as the BT DSL?](../getting-oriented/why-lisp-dsl.md).

The baseline evidence includes:

- representative DSL round-trip checks for sequence, selector, reactive, planner, and async/VLA-shaped BTs
- `bt.to-dsl`, `bt.save-dsl`, and `bt.load-dsl` coverage through deterministic tests
- source and canonical DSL hash support for DSL-backed `bt_def` events
- negative generated-fragment fixtures for:

    - unknown node type
    - unknown host callback
    - unsupported capability
    - invalid budget
    - malformed subtree
    - missing fallback around a long-running async/model call

## capability-event decision

`v0.7.0` records the decision for future real host capability adapters:

- real host capability calls that can affect runtime behaviour should emit canonical lifecycle events
- the chosen user-facing event shape is `cap_call_start` and `cap_call_end`
- those events should carry request id, capability, operation, status, latency, rejection/error reason where applicable, and enough tick/job context for replay and first-divergence reports
- deterministic `cap.echo.v1` remains a registry/API smoke path without extra canonical capability lifecycle events until it stops being a pure fixture

## compatibility notes

- previous runtime contract version: `1.0.0`
- new runtime contract version: `1.0.0`
- previous event schema: `mbt.evt.v1`
- new event schema: `mbt.evt.v1`
- compatibility impact: additive runtime metadata, docs, tests, and fixtures

Migration notes for consumers:

- existing BT programs do not need to change
- existing event consumers should tolerate additive `bt_def` payload fields:

    - `source_hash`
    - `canonical_dsl_hash`

- `tree_hash` now uses the canonical DSL hash for DSL-backed definitions when available, with a structural fallback for non-DSL definitions
- generated-fragment validation fixtures are evidence and tooling for the DSL boundary; they do not make generated fragments executable by default

## what is intentionally limited

- `muesli-bt` remains a task-level runtime, not a hard real-time servo-loop controller
- the Lisp DSL is not a general-purpose plugin language for arbitrary robot IO
- generated BT fragments are treated as untrusted until parsed, normalised, validated, checked against capability and fallback rules, logged, and replayed
- generated subtree execution remains later roadmap work
- `cap_call_start` and `cap_call_end` are the chosen event direction for real capability adapters, not yet a released robot adapter surface
- `cap.echo.v1` remains a deterministic fixture capability

## release posture

`v0.7.0` is a core-defensibility and async-correctness release.

The main change is evidence quality. The runtime now has a clearer evidence story for async cancellation, canonical lifecycle logs, DSL round-tripping, generated-fragment rejection, and BT definition identity in logs.

# v0.8.0 draft support boundary

## what this is

This page records the intended `v0.8.0` support boundary before the release is tagged.

Use the latest tagged release note for stable released behaviour. Use this page to review the current `main` branch candidate surface.

## when to use it

Use this page when you need to decide whether a model-service, VLA, replay, or evidence-path change belongs in the `v0.8.0` release candidate.

## how it works

`v0.8.0` is the model-backed runtime hardening milestone. The release candidate keeps core BT and Lisp semantics unchanged while adding an optional model-service bridge and evidence path.

The bridge is optional and disabled by default. `muesli-bt` must build, test, and run without `muesli-model-service` unless a host explicitly configures model capabilities.

## api / syntax

Candidate support boundary:

| Surface | Boundary |
| --- | --- |
| Core BT and Lisp semantics | unchanged from the released runtime contract |
| Optional model-service bridge | experimental `MMSP v0.2` candidate, build-time optional |
| No-service behaviour | supported: no-model BTs must not contact or require the service |
| `describe` compatibility checks | candidate support for protocol version, descriptor shape, mode, schema, cancellation, deadline, freshness, and replay declarations |
| Stateless model calls | experimental `cap.call` path for `cap.model.world.rollout.v1` and `cap.model.world.score_trajectory.v1` |
| VLA sessions | experimental model-service backend for `cap.vla.action_chunk.v1` through `vla.submit`, `vla.poll`, and `vla.cancel` |
| Frame ingest | documented HTTP ingest plus `frame://` refs; immutable refs are required for release evidence |
| Hashing and replay | candidate support for request/response hashes and request-hash keyed replay cache for stateless calls and VLA sessions |
| Validation gates | candidate support for rejecting malformed, stale, unsafe, late, or policy-violating proposals before host reach |
| Fault injection | candidate deterministic schedules for stateless model calls and VLA sessions |
| MiniVLA evidence | curated gated evidence path using checked-in Webots frames and release-safe summaries, run against `muesli-model-service v0.2.0` |
| Mock-host handoff | evidence-only proof that accepted proposals can cross a mock host boundary |
| Physical robot dispatch | not released in `v0.8.0` |
| Production VLA provider support | not released in `v0.8.0` |
| Nav2 and MoveIt adapters | not released in `v0.8.0` |

## example

For release evidence, use the curated MiniVLA path rather than ad hoc images:

```bash
python3 tools/evidence/run_minivla_smoke.py \
  --http-endpoint http://127.0.0.1:8765 \
  --ws-endpoint ws://127.0.0.1:8765/v1/ws \
  --service-release-tag v0.2.0 \
  --service-git-commit be3ff75
```

The current release-candidate evidence uses `muesli-model-service v0.2.0` at commit `be3ff75`.
The generated `manifest.json` must record that service tag and commit.

Expected release-safe outputs include:

- `release_safe_replay_report.json`
- `release_safe_frame_refs.json`
- `release_safe_cache_summary.json`
- `release_safe_model_service_describe.json`
- `release_safe_prompt_summary.json`
- `redaction_policy.json`

## gotchas

- The mock-host report is not physical robot evidence.
- `frame://camera1/latest` is acceptable for live operation, but release evidence should record immutable frame refs.
- Raw prompts, raw frame refs, raw model-service envelopes, request ids, session ids, and backend placement metadata belong in private reproduction bundles unless reviewed.
- `MMSP v0.2` `status` operation coverage is only required for the tag if a supported backend needs it.
- Production provider reliability and robot safety remain host responsibilities outside this candidate release boundary.

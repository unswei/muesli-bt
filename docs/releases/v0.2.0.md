# v0.2.0 release notes

Release date: 2026-03-04

## what is stable in v0.2.0

- runtime contract documentation and compatibility policy:

    - `docs/contracts/runtime-contract-v1.md`
    - `docs/contracts/compatibility.md`
    - `docs/contracts/conformance.md`

- canonical event stream surface:

    - schema name `mbt.evt.v1`
    - published schema path `schemas/event_log/v1/mbt.evt.v1.schema.json`
    - event envelopes include `contract_version`

- deterministic conformance and fixture tooling:

    - `tests/conformance/` `L0` suite
    - `tools/fixtures/update_fixture.py`
    - `tools/fixtures/verify_fixture.py`
    - canonical validator `tools/validate_log.py`

- release packaging:

    - source archives from the release tag
    - Ubuntu `x86_64` binary archive
    - macOS `arm64` binary archive

## compatibility notes

- previous runtime contract version: `1.0.0`
- new runtime contract version: `1.0.0`
- previous event schema: `mbt.evt.v1`
- new event schema: `mbt.evt.v1`
- compatibility impact: additive

Migration notes for consumers:

- tolerate `contract_version` in canonical event envelopes and `run_start` metadata
- expect additional runtime-contract events such as `node_enter`, `node_exit`, `budget_warning`, `deadline_exceeded`, `planner_call_start`, `planner_call_end`, `async_cancel_requested`, `async_cancel_acknowledged`, and `async_completion_dropped`
- prefer the published schema path under `schemas/event_log/v1/`
- use the conformance fixtures and verifier tooling as the release-aligned compatibility check

## what is intentionally limited

- ROS2 in `v0.2.0` is still a contract and CI-preparation surface, not a supported real transport baseline
- `L2` remains a future ROS2-focused conformance lane, not a deployability claim for this tag
- simulator demos remain the main runnable integration path

## release posture

`v0.2.0` is the first release that locks the runtime contract, conformance framing, and release packaging into one coherent baseline for downstream consumers.

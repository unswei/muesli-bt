# v0.3.0 release notes

Release date: 2026-03-14

## what is stable in v0.3.0

- runtime contract and event schema remain stable for downstream consumers:

    - `docs/contracts/runtime-contract-v1.md`
    - `docs/contracts/compatibility.md`
    - `schemas/event_log/v1/mbt.evt.v1.schema.json`

- conformance framing and runbooks remain stable:

    - `docs/contracts/conformance.md`
    - `L0` core conformance in CI
    - `L1` simulator conformance in CI

- the first ROS2 thin-adaptor release baseline is now part of the supported surface:

    - Ubuntu 22.04
    - ROS 2 Humble
    - `(env.attach "ros2")`
    - `nav_msgs/msg/Odometry` input
    - `geometry_msgs/msg/Twist` output
    - `muesli_bt::integration_ros2` exported for consumers
    - `muslisp_ros2` live runner

- ROS2 release evidence now includes:

    - Linux ROS-backed test coverage
    - rosbag-backed `L2` replay artefact verification
    - installed-package export plus ROS2 consumer smoke
    - live runner validation with `examples/repl_scripts/ros2-live-odom-twist.lisp`

## compatibility notes

- previous runtime contract version: `1.0.0`
- new runtime contract version: `1.0.0`
- previous event schema: `mbt.evt.v1`
- new event schema: `mbt.evt.v1`
- compatibility impact: additive

Migration notes for consumers:

- ROS2 consumers can now treat the thin Ubuntu 22.04 + Humble `Odometry` -> `Twist` path as a released baseline rather than a pre-release contract surface
- use `muesli_bt::integration_ros2` and `muslisp_ros2` instead of treating the ROS2 path as an internal or ad hoc build target
- expect `backend_version`, `obs_schema`, `state_schema`, and `action_schema` on the ROS2 backend to stay within the current `ros2.*.v1` families
- if you are building tooling, keep using the canonical `mbt.evt.v1` schema and release notes together with the Studio compatibility matrix
- the attached Ubuntu binary archive for `v0.3.0` is the generic non-ROS package; ROS2 remains a released source/install baseline at this tag rather than a separate ROS-enabled binary artefact

## what is intentionally limited

- ROS2 support in `v0.3.0` is intentionally narrow:

    - Ubuntu 22.04 + Humble only
    - `Odometry` in, `Twist` out
    - no released MoveIt, Nav2, perception, or ROS action/service integration surface

- live ROS-backed runs still rely on run-loop artefact JSONL for the current replay/logging flow; direct canonical `mbt.evt.v1` parity for ROS-backed runs remains `v0.4.0` work
- live reset stays unsupported; `reset_mode="stub"` remains for tests and harnesses only

## release posture

`v0.3.0` is the first release that treats the ROS2 thin adaptor as a supported release surface rather than a preparatory contract lane.

The main release claim is deliberately modest:

- same runtime semantics
- same `env.*` attach/configure surface
- different host transport through ROS2

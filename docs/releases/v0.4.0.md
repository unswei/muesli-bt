# v0.4.0 release notes

Release date: 2026-03-21

## what is stable in v0.4.0

- runtime contract and canonical event schema remain unchanged for downstream consumers:

    - `docs/contracts/runtime-contract-v1.md`
    - `docs/contracts/compatibility.md`
    - `schemas/event_log/v1/mbt.evt.v1.schema.json`

- the released ROS2 thin-adaptor baseline remains intentionally narrow:

    - Ubuntu 22.04
    - ROS 2 Humble
    - `(env.attach "ros2")`
    - `nav_msgs/msg/Odometry` input
    - `geometry_msgs/msg/Twist` output
    - `muesli_bt::integration_ros2`
    - `muslisp_ros2`

- canonical observability and replay tooling now cover ROS-backed and simulator-backed runs through the same artefact shape:

    - `event_log_path` writes direct canonical `mbt.evt.v1` JSONL
    - artefact directories use `events.jsonl`
    - `tools/validate_log.py` accepts either a direct log file or an artefact directory
    - `tools/validate_trace.py` validates cross-event invariants and replay comparisons

- canonical lifecycle and timing metadata are now explicit in released ROS-backed logs:

    - `episode_begin`
    - `episode_end`
    - `run_end`
    - `time_source`
    - `use_sim_time`
    - `obs_timestamp_source`

- release artefacts remain split by support surface:

    - generic Ubuntu `x86_64` archive (non-ROS)
    - Ubuntu 22.04 `x86_64` archive with ROS 2 Humble support
    - macOS `arm64` archive

## compatibility notes

- previous runtime contract version: `1.0.0`
- new runtime contract version: `1.0.0`
- previous event schema: `mbt.evt.v1`
- new event schema: `mbt.evt.v1`
- compatibility impact: additive tooling and observability clarification

Migration notes for consumers:

- tooling consumers can now treat ROS-backed and simulator-backed runs through the same canonical `events.jsonl` path
- use both validators together: `tools/validate_log.py` for per-record schema checks and `tools/validate_trace.py` for cross-event validation and replay comparison
- if you read ROS-backed canonical logs, expect the expanded lifecycle anchors (`episode_begin`, `episode_end`, `run_end`) and ROS time-source metadata in `run_start.data.capabilities`
- no parser, schema, or runtime-contract migration is required from `v0.3.1`

## what is intentionally limited

- the ROS-enabled release artefact is still tied to the first supported ROS lane:

    - Ubuntu 22.04
    - ROS 2 Humble
    - `Odometry` in, `Twist` out

- live reset remains unsupported; `reset_mode="stub"` remains for tests and harnesses only
- this release does not broaden ROS transport scope beyond the first thin adaptor
- production VLA/provider integration remains host-owned rather than a built-in runtime transport surface

## release posture

`v0.4.0` is the observability-parity and replay-verification release.

The runtime contract does not move.
The main change is that ROS-backed runs now participate in the same canonical event-log and replay story as simulator-backed runs, with released validation tooling and explicit timing/lifecycle metadata.

# why Lisp as the BT DSL?

`muesli-bt` uses Lisp because Behaviour Trees are tree-structured task logic, and Lisp S-expressions are compact structured tree data.

This is a technical choice about representation. It is not a claim that Lisp is generally better than C++, XML, YAML, Python, or any other BT authoring format.

## what this is

This page explains the role of Lisp in `muesli-bt`.

The Lisp layer is the task-level Behaviour Tree (BT) representation. It gives the runtime a form that can be read by humans, generated by tools, checked by validators, serialised for artefacts, and replayed later.

## when to use it

Read this page if you want to understand why the project uses Lisp forms such as:

```lisp
(defbt safe-goal
  (sel
    (seq
      (cond obstacle-clear)
      (act move-towards-goal))
    (act stop)))
```

The short version is:

- Behaviour Trees are trees.
- Lisp S-expressions are structured tree data.
- A Lisp BT DSL can therefore be both source code and inspectable runtime data.

## how it works

A BT DSL form is parsed and checked before it becomes a runtime `bt_def`.

The same representation can serve several roles:

- hand-authored BT source;
- generated BT fragment;
- validation input;
- canonical serialisation format;
- replay artefact;
- safe runtime patch format.

This matters for robotics because task logic often needs to be inspected, compared, generated, or transformed without becoming opaque host code.

For example, a future generated recovery subtree can be represented as Lisp data, normalised to a canonical form, validated against the BT grammar and host capability contract, compiled to a `bt_def`, logged by hash, installed at a tick boundary, and replayed from the same artefact.

## safety boundary

The Lisp DSL owns task-level decision structure. It does not own robot IO, safety-critical servo loops, or hard real-time control.

Host capabilities own those boundaries:

- robot drivers;
- ROS2, Nav2, MoveIt, or simulator integration;
- actuator commands;
- sensor transport;
- safety-critical low-level control.

Generated Lisp fragments must be treated as untrusted input until they pass the runtime checks. A fragment is not safe because it is Lisp. It is only eligible to run after parsing, normalisation, validation, capability checks, budget checks, and fallback checks.

## api / syntax

The defensible path is:

```text
Lisp BT DSL
    -> parsed form
    -> normalised form
    -> validated BT grammar and host contracts
    -> compiled bt_def
    -> canonical DSL artefact and hash
    -> tick-boundary installation
    -> canonical event log
    -> replay / comparison
```

The runtime contract should be able to record:

- which BT definition or fragment was used;
- which source or canonical DSL hash identified it;
- when validation happened;
- whether validation passed or failed;
- when installation happened;
- how the tree behaved during replay.

These hashes are identity and replay aids, not a safety proof.

- `source_hash` identifies the source DSL text supplied to `bt.compile` or `bt.load-dsl`.
- `canonical_dsl_hash` identifies the normalised DSL returned by `bt.to-dsl`.
- `tree_hash` uses the canonical DSL hash when available, with a structural fallback for non-DSL definitions.

Use the source hash to answer "which generated or loaded fragment was proposed?". Use the canonical DSL hash to answer "which normalised tree was accepted, logged, and replayed?". If the source hash changes but the canonical DSL hash stays the same, the difference was normalised away. If the canonical DSL hash changes, replay and paper artefacts can treat it as a different BT definition.

## example

The minimal example above is hand-authored, but it has the same tree shape a generator could produce:

```lisp
(sel
  (seq
    (cond obstacle-clear)
    (act move-towards-goal))
  (act stop))
```

The important point is that this is not arbitrary host-side robot code. It is a small tree with known node types, known leaf references, and a shape that the runtime can validate before execution.

The `v0.7.0` baseline is to make this representation argument explicit and testable through round-trip checks, canonical hashes where practical, and deterministic rejection fixtures for invalid generated fragments.

## current round-trip evidence

The `muslisp_tests` suite includes a deterministic representative round-trip check named `bt representative dsl roundtrip shapes`.

It checks this path for each representative shape:

```text
source DSL -> parsed form -> compiled bt_def -> canonical DSL -> parsed form -> compiled bt_def
```

It also saves and reloads the canonical DSL artefact with `bt.save-dsl` and `bt.load-dsl`.

The same suite includes `bt dsl hashes logged for compiled and loaded definitions`, which confirms that DSL-backed `bt_def` events identify compiled and loaded definitions with `source_hash` and `canonical_dsl_hash` where the source text is available.

The negative generated-fragment fixtures under `fixtures/dsl/generated-fragment-negative` check deterministic rejection before execution for:

- unknown node type;
- unknown host callback;
- unsupported host/model capability;
- invalid budget;
- malformed subtree shape;
- missing fallback around a long-running async/model call.

| shape | representative coverage | current expected result |
| --- | --- | --- |
| sequence | `seq`, `act`, `cond` | pass |
| selector | `sel`, `cond`, `succeed` | pass |
| reactive | `reactive-sel`, `reactive-seq`, `async-seq` | pass |
| planner | `plan-action` with budget and model-service metadata | pass |
| async / VLA-shaped | `vla-request`, `vla-wait`, `vla-cancel` | pass |

Run the check with:

```bash
cmake --build --preset dev -j
ctest --preset dev -R muslisp_tests --output-on-failure
python3 tools/validate_generated_bt_fragment.py fixtures/dsl/generated-fragment-negative
```

This is representative DSL evidence. It does not prove that every generated BT fragment is safe. Generated fragments still need grammar validation, capability checks, budget checks, fallback checks, canonical logging, and replay evidence before execution.

The later paper-facing demonstration is a generated guarded recovery subtree. That demonstration is planned to show the full path: generate as Lisp data, normalise, validate, compile, serialise, install at a tick boundary, execute, log, and replay.

## gotchas

- Do not present Lisp as a virtue by itself. The argument is structured tree data for BTs.
- Do not execute raw model-generated text as robot control logic.
- Do not bypass the same runtime contract, canonical logging, or replay validation used by hand-authored BTs.
- Do not install generated subtrees in the middle of a tick.
- Do not claim that Lisp makes runtime task generation safe by itself.
